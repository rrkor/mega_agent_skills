# KB (База знаний, БЗ) и RAG

Этот файл покрывает tool-переменные:
- `list_kbs`
- `get_kb`
- `create_kb`
- `update_kb`
- `delete_kb`
- `list_documents`
- `get_document`
- `delete_document`
- `search_info_from_kb`

---

## `list_kbs` — получить список баз знаний
 
**Destructive:** `false`

### Переменные
- не нужны

### Когда использовать
Используется, чтобы получить обзор всех баз знаний внутри проекта.

Особенно полезен в таких случаях:
- перед созданием новой базы знаний, чтобы проверить, нет ли уже существующей KB с похожим или таким же названием;
- перед обновлением или удалением KB, если у пользователя нет точного `kb_id`;
- когда пользователь формулирует задачу через название, а не через идентификатор;
- перед поиском по KB, если сначала нужно понять, в какой именно базе работать;
- в ситуациях, где нужно показать пользователю доступные базы знаний или выбрать одну из нескольких.

Для planner-а это стартовый шаг, когда задача связана с KB, но нужная база ещё не определена.

## `get_kb` — получить одну базу знаний

**Destructive:** `false`

### Переменные
- `kb_id` - ID базы знаний, НЕ НАЗВАНИЕ! (обяз.)

### Когда использовать
Используй этот tool, когда база знаний уже выбрана и нужно получить её детальную информацию.

Подходит для таких задач:
- проверить свойства конкретной KB перед изменением;
- убедиться, что выбрана правильная база знаний;
- получить подробности по настройкам KB;
- использовать как промежуточный шаг перед обновлением или удалением;
- проверить состояние KB после создания или изменения.

Для planner-а это шаг уточнения: не обзор всех KB, а работа с уже известной конкретной базой.

## `create_kb` — создать базу знаний (БЗ)
  
**Destructive:** `false`

### Переменные
- имя базы (`name`) (обяз.)
- `sourceType` (`upload`, `confluence`) (обяз.). Если пользователь не уточняет источник - ставим `upload`

### Когда использовать
Используй этот tool, когда задача пользователя подразумевает создание новой базы знаний.

Подходит для таких ситуаций:
- пользователь явно просит создать KB;
- нужно завести новую БЗ, базу знаний, knowledge base или RAG-хранилище;
- нужно подготовить отдельную базу под документы, справку, инструкции, аналитику или retrieval;
- нужно создать KB с конкретными параметрами чанкинга или embedding.

Для planner-а важно:
- не создавать KB вслепую, если возможно, сначала полезно проверить текущий список KB через `list_kbs`;
- если пользователь не назвал базу, может потребоваться уточнение;
- если имя уже занято или есть похожая KB, стоит сначала проверить существующие базы.

## `update_kb` — обновить базу знаний

**Destructive:** `false`

### Переменные
- `kb_id` - ID базы знаний (обяз.)
- `name` - имя базы знаний (обяз.)

### Когда использовать
Используй этот tool, когда нужно изменить свойства уже существующей базы знаний.

Подходит для таких задач:
- переименовать KB;
- обновить метаданные KB;
- привести название базы в соответствие с новым назначением;
- выполнить точечное изменение уже существующей базы.

Для planner-а важно:
- сначала убедиться, что выбрана правильная KB;
- не использовать `update_kb`, если пользователь на самом деле хочет создать новую KB;
- если пользователь указывает название, а не `kb_id`, сначала может понадобиться `list_kbs`.

---

## `delete_kb` — удалить базу знаний

**Destructive:** `true`

### Переменные
- `kb_id` - использовать только ID базы знаний, НЕ НАЗВАНИЕ! (обяз.)

### Когда использовать
Используй этот tool только тогда, когда пользователь явно хочет удалить конкретную базу знаний.

Подходит для таких ситуаций:
- пользователь просит удалить KB;
- нужно убрать ненужную или ошибочно созданную базу;
- требуется очистить проект от конкретной KB.

Для planner-а важно:
- используется именно ID базы знаний, а не ее название
- если пользователь указывает название, а не `kb_id`, сначала лучше определить точную базу через `list_kbs` или `get_kb`;
- это деструктивное действие;
- нельзя удалять KB, если неясно, какая именно база нужна;
- если в проекте есть несколько похожих баз, лучше не делать удаление без дополнительной уверенности.

## `list_documents` — получить документы определенной KB

**Destructive:** `false`

### Переменные
- `kb_id` - использовать только ID базы знаний, НЕ НАЗВАНИЕ! (обяз.)

### Когда использовать
Используй этот tool, когда нужно посмотреть состав документов внутри базы знаний.

Подходит для таких задач:
- проверить, какие документы уже загружены в KB;
- понять, есть ли нужный документ в базе;
- подготовиться к удалению документа;
- убедиться, что база не пустая;
- проверить результат после загрузки документов;
- выбрать конкретный документ для дальнейшего просмотра или удаления.

Для planner-а это основной шаг, если задача относится не ко всей KB, а к её содержимому.

---

## `get_document` — получить документ

**Destructive:** `false`

### Переменные
- `kb_id` - использовать только ID базы знаний, НЕ НАЗВАНИЕ! (обяз.)
- `document_id`

### Когда использовать
Используй этот tool, когда документ уже известен и нужно получить именно его.

Подходит для таких задач:
- открыть конкретный документ;
- проверить свойства документа;
- убедиться, что найден правильный документ перед удалением;
- получить подробности по одному документу после `list_documents`.

Для planner-а важно:
- если `document_id` ещё неизвестен, сначала обычно нужен `list_documents`;
- этот tool не подходит для обзора всей KB, только для точечной работы с одним документом.

---

## `delete_document` — удалить документ из базы знаний

**Destructive:** `true`

### Переменные
- `kb_id` - использовать только ID базы знаний, НЕ НАЗВАНИЕ! (обяз.)
- `document_id` - ID документа, НЕ НАЗВАНИЕ! (обяз.)

### Когда использовать
Используй этот tool, когда пользователь явно хочет удалить конкретный документ из базы знаний.

Подходит для таких ситуаций:
- удалить ненужный документ;
- очистить KB от устаревшего файла;
- убрать ошибочно загруженный документ.

Для planner-а важно:
- это деструктивное действие;
- нельзя удалять документ, если не установлен точный `document_id`;
- если пользователь называет документ словами, а не ID, сначала лучше получить список документов и выбрать нужный;
- если в KB несколько похожих документов, нужно быть особенно осторожным.

## `search_info_from_kb` — поиск по knowledge base

**Destructive:** `false`

### Переменные:
- `kb_id` - использовать только ID базы знаний, НЕ НАЗВАНИЕ! (обяз.)
- `query` (обяз.)
- `topK` (обяз.)

### Когда использовать
Используй этот tool, когда задача связана не с управлением KB как объектом, а с извлечением знаний из её содержимого.

Подходит для таких сценариев:
- выполнить семантический поиск по документам KB;
- найти ответ на вопрос внутри базы знаний;
- получить релевантные куски
- проверить, есть ли в KB информация по теме пользователя;
- использовать KB как внутренний источник знаний для планирования, проверки или генерации ответа.

Для planner-а это ключевой tool retrieval-логики.
Он нужен, когда цель — не изменить KB, а извлечь из неё смысловую информацию.

Используй `search_info_from_kb`, когда:
- у пользователя вопрос по содержимому документов;
- нужно “заглянуть” в knowledge base;
- важно найти релевантные фрагменты, а не просто перечислить документы.
